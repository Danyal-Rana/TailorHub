ROLE 1: FULL-STACK ENGINEER
================================================================

Full-stack bullets lead with backend/system-level complexity (60%),
add frontend bullets only where they show engineering depth.

PATTERN 1 — OUTCOME FIRST:

- Reduced Firestore order-list read cost to 0 on cache hits by implementing
  a Redis TTL cache (60s, 4 key patterns) via server-side Next.js Route
  Handlers, cutting repetitive reads for the 2 highest-volume roles on
  every /orders page load.

- Delivered new notifications to clients in ≤2s without long-polling by
  engineering a dual-channel push system — Firestore onSnapshot (persistent
  WebSocket) + Redis LIST polled over SSE every 2s — with ID-based client
  deduplication preventing double-render across the 2 delivery channels.

- Enforced zero-bypass RBAC across all 8 Firestore collections by authoring
  Security Rules that independently verify uid, role, and status on every
  read/write — ensuring client-side guard bypasses cannot reach raw data.

PATTERN 2 — STAR (Problem → Engineering Action → Result):

- Resolved a Timestamp serialization failure between Firestore objects and
  Redis JSON storage by building a bidirectional conversion layer:
  serializeOrder() converts Timestamps to ISO-8601 on cache write,
  deserializeOrder() reconstructs .toDate()-compatible proxies on read —
  preserving the interface contract of all 3 downstream UI components.

- Resolved Upstash credential exposure risk in a client-rendered app by
  proxying all 3 Redis operation types (read/write/invalidate) through
  server-side Route Handlers — ensuring the Upstash REST token is absent
  from every browser JavaScript bundle.

- Resolved SSE disconnection on Vercel's 25s serverless timeout by
  designing bounded-session streaming with EventSource.onerror reconnect
  (1s delay) on the client — maintaining uninterrupted notification delivery
  across session resets with no user-visible gap.

PATTERN 3 — WWH (What / Why / How — Architecture):

- Architected a 7-stage order state machine (pending → measurement_needed →
  in_progress → ready → out_for_delivery → delivered → cancelled) with per-
  transition cache invalidation and customer notification dispatch — maintaining
  data consistency across 3 concurrent user roles modifying the same Firestore
  order document.

- Engineered a Timestamp-safe Redis caching pipeline to bridge the type
  mismatch between Firebase Timestamp objects and JSON-serializable Redis
  values, using ISO-8601 as intermediate format and a .toDate()-proxy shim on
  deserialization — enabling cache-hit responses with 0 component-layer changes.

- Built 3-provider Firebase Authentication (Email/Password, Google OAuth,
  Phone SMS via RecaptchaVerifier) with a unified Firestore AppUser type
  to consolidate identity across 4 user roles regardless of sign-in method,
  using a single onAuthStateChanged listener.

PATTERN 4 — SCALE & SCOPE:

- Architected a full-stack platform supporting 4 user roles, 8 Firestore
  collections, 5 REST API routes, 3 Firebase auth providers, 4 external
  integrations, 7 order status stages, and 7 notification event types —
  deployed to Vercel with 18 env vars across 2 access tiers.

- Engineered a notification system spanning 7 event types, 2 delivery channels
  (Firestore WebSocket + Redis SSE), per-user LIST queues (1h TTL), and
  client-side deduplication across 50 most-recent notifications per user.

- Built a multi-role measurement module supporting 3 dress types, 12 fields
  (9 kameez-specific with Urdu labels, 3 shalwar-specific), versioned records
  per customer, and a 4-stage measurement request workflow.

ROLE 2: BACKEND ENGINEER
================================================================

Backend bullets go deep on data modeling, API design, auth, caching,
validation. Frontend is invisible in this section.

PATTERN 1 — OUTCOME FIRST:

- Eliminated repeated Firestore list query cost by designing a 4-key Redis
  caching strategy (orders:all, orders:customer:{uid}, orders:delivery:{uid},
  60s TTL) with targeted key deletion on createOrder and updateOrderStatus —
  preventing stale reads while avoiding full-cache flushes on unrelated writes.

- Enforced data access control for 8 Firestore collections by authoring
  Security Rules with per-role, per-status conditions checking request.auth.uid
  — eliminating server-bypass attack vectors that client-side guards cannot
  block.

- Guaranteed notification fan-out to all active staff (admin + tailor) in
  parallel by querying recipient UIDs by role and dispatching Firestore writes
  via Promise.all — avoiding N serial round-trips on every new-order event.

PATTERN 2 — STAR:

- Resolved a type-system gap where Firestore Timestamp objects fail JSON
  serialization to Redis by implementing serializeOrder() / deserializeOrder()
  utilities that convert to ISO-8601 for cache storage and reconstruct
  .toDate()-compatible proxy objects on read — with 0 changes to the 3
  consuming UI components.

- Resolved a dual-enforcement gap where role checks existed client-side only
  by authoring Firestore Security Rules with 8 match blocks covering every
  collection — creating 2 independent RBAC layers where bypassing one does
  not bypass the other.

- Resolved notification delivery coupling between creation latency and UI
  responsiveness by fire-and-forgetting a Redis LIST publish immediately after
  Firestore write — allowing the SSE stream to deliver in ≤2s while Firestore
  propagation completes asynchronously.

PATTERN 3 — WWH:

- Modeled an 8-collection Firestore schema with 5 cross-collection reference
  fields (customerId, linkedUserId, orderId, recipientUid, assignedTailor)
  to support 4 user roles with distinct read scopes — avoiding JOIN overhead
  by denormalizing role-specific foreign keys into order and delivery documents.

- Designed a 5-endpoint internal API surface using Next.js Route Handlers as
  a server-side Redis proxy, implementing the full cache lifecycle: GET (read),
  POST (write with 60s TTL), DELETE (targeted key invalidation) — keeping all
  Upstash credentials server-side only.

- Engineered a 3-provider auth architecture (Email/Password, Google OAuth,
  Phone SMS) with a unified AppUser Firestore document and a 2-state approval
  workflow (pending_approval → active) for tailor and delivery accounts
  requiring admin sign-off before system access.

PATTERN 4 — SCALE & SCOPE:

- Engineered a backend service layer across 8 TypeScript modules (orderService,
  notificationService, customerService, measurementService, adminService,
  deliveryService, inventoryService, userService) with 4 external integrations
  and Zod-validated input at all form boundaries.

- Architected a notification data model supporting 7 event types, 2 delivery
  channels, per-user LIST queues (1h TTL), and Security Rules restricting
  isRead mutations to the recipientUid only — enforcing integrity at the
  infrastructure layer, not the application layer.

- Implemented a multi-role query engine dispatching 3 distinct Firestore
  query strategies (unconstrained for admin/tailor, customerUid-filtered for
  customer, deliveryBy-filtered for delivery) with 4-key Redis caching and
  targeted invalidation on 2 mutation operations.

HONEST ASSESSMENT
================================================================

WHAT'S STRONG (top 3 resume-worthy signals):

1. DUAL-LAYER RBAC ARCHITECTURE
   Firestore Security Rules + client-side RoleGate/AuthGate = 2 independent
   enforcement layers. Most student projects have one or the other. Authoring
   Security Rules correctly across 8 collections with uid/role/status checks
   is a real backend maturity signal that survives technical recruiter review.

2. REDIS CACHING WITH DELIBERATE INVALIDATION DESIGN
   Not just "added Redis" — designed a 4-key cache schema, chose 60s TTL
   deliberately, wired targeted per-key invalidation on 2 specific mutation
   operations, and built a Timestamp serialization bridge for the Firestore-
   to-JSON impedance mismatch. This is systems thinking, not tutorial-following.

3. DUAL-CHANNEL REAL-TIME NOTIFICATION SYSTEM
   Combining Firestore onSnapshot (reliability) with Redis LIST + SSE
   (latency) and handling ID-based deduplication client-side is a non-trivial
   architectural decision. It demonstrates awareness of the reliability-vs-
   latency trade-off that FAANG interviewers probe in system design rounds.

WHAT'S MISSING (top 3 gaps):

1. NO CI/CD PIPELINE [45-minute fix]
   Zero GitHub Actions. A single workflow (lint → typecheck → Vercel preview
   deploy) immediately adds 4 DevOps bullets with real stage counts. Without
   it, the DevOps role is a hard sell to any technical recruiter screening
   for pipeline experience.

2. NO LLM INTEGRATION [2-4 hour fix]
   The Gen AI role has zero direct evidence. Adding 1 Anthropic API call
   (e.g., auto-generate order notes from fabric + customer context, streamed
   via SSE) creates genuine Gen AI bullets around prompt engineering, server-
   side key isolation, and streaming output — and the streaming infrastructure
   already built here maps directly to that implementation.

3. NO OBSERVABILITY / ERROR TRACKING [30-minute fix]
   No Sentry, no logging, no performance monitoring. Adding Sentry (free tier)
   gives a real "reduced MTTR" bullet. Without observability, the system has
   no operational story — which matters for Backend + Infra and DevOps roles
   specifically, where recruiters expect at least basic error tracking.

WHICH ROLE DOES THIS CODEBASE SUPPORT BEST:

Full-Stack Engineer, closely followed by Backend Engineer. The evidence is
genuine: an 8-collection Firestore data model with cross-collection references,
a 4-role RBAC system enforced at 2 independent layers, a Redis caching layer
with deliberate TTL and invalidation design, a dual-channel notification system
with architectural reasoning behind both channels, a Timestamp serialization
bridge between Firebase and JSON storage, and 3-provider auth integration with
a role-specific approval workflow. None of these are boilerplate CRUD — each
represents an engineering decision with visible trade-offs. A recruiter reading
the bullets alongside the actual codebase will find the story consistent and
defensible. The 4 distinct role-specific dashboards also round out the full-
stack narrative with real multi-role UX complexity.

WEAKEST EVIDENCE PER ROLE:

- Full-Stack: Strong — add 1 frontend perf bullet (lazy loading, Suspense, or
  bundle analysis) to complete the story beyond backend depth.

- Backend: Strong — run a k6 or Autocannon load test to get real p95 latency
  numbers; current latency claims (≤2s) are architectural estimates, not

- Backend + Infra: Moderate — add a Dockerfile (multi-stage Node.js build) and
  this section immediately becomes strong; without containers the infra story

- DevOps / Cloud: Weak — no CI/CD is the critical gap; a 3-stage GitHub
  Actions workflow (lint, typecheck, deploy) doubles bullet count and adds the
  1 thing DevOps recruiters screen for first.

- Gen AI: No evidence — zero AI components in this codebase; add any LLM API
  call (Anthropic messages API, streamed via the SSE infrastructure already
  built here) to convert analogy into evidence.
